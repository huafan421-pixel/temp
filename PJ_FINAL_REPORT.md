# 小学数学应用题自动解题项目实验报告

## 1. 项目背景与任务理解

本次 PJ 面向“小学数学应用题自动解题”任务。给定一段小学 1-6 年级数学应用题文本，系统需要输出最终数值答案，提交文件为两列 CSV，格式为 `id,ret`，其中 `ret` 只保留数字、分数或百分数等答案本体，不包含单位和解释。训练集包含约 11999 条带答案样本，测试集包含 8000 条题目。本项目的目标是在满足课程约束的前提下，构建一个可以自动生成提交文件的解题系统，并通过平台提交获得客观分数。

课程要求中对模型规模和数据来源有明确限制：推理模型应控制在 0.5B 参数量或更小；数据增强、监督微调、提示词样例和规则构建都应基于给定训练集完成，不使用外部标注答案。基于这一要求，本项目选择 `Qwen2.5-0.5B-Instruct` 作为主要小模型，并围绕它完成 prompt 推理、LoRA SFT 微调、CoT 数据构建等探索。同时，由于小学数学题具有大量固定题型和可解析的算术结构，项目还实现了一个高置信规则/模板求解器，并最终将模型输出、规则校验和训练集同题/同模板校正融合，形成最终提交方案。

本项目最终采用的提交文件为：

```text
submit_fewshot_improved_rule_hybrid_v6_memory.csv
```

该版本在平台公开榜上的得分为：

```text
0.194625
```

对应 8000 条测试题中约 1557 条答对。与初始 few-shot CoT 方案的 `0.182125` 相比，最终方案净提升 `0.0125`，约等于净多答对 100 道题。

## 2. 总体方案设计

本项目没有只依赖单一路线，而是完成了多种互补方案的实验探索。整体思路可以概括为：

1. 使用 0.5B 小模型直接生成答案，评估 prompt 设计、zero-shot CoT 和 few-shot CoT 的效果。
2. 使用训练集构造增强数据和 CoT 格式数据，对 Qwen2.5-0.5B-Instruct 做 LoRA SFT。
3. 针对小学数学高频题型编写规则求解器，用于纠正常见算术错误。
4. 利用训练集中与测试集完全相同或高度相似的题型，构建高置信答案校正层。
5. 对所有候选 CSV 做严格格式校验和逐步提交，依据平台分数筛选最终方案。

最终系统不是简单的模型投票，而是一个“模型基础答案 + 高置信规则覆盖 + 训练集模板记忆”的混合系统。这样设计的原因是：在 0.5B 规模下，小模型具有一定语言理解能力，但算术稳定性不足，尤其容易在多步应用题中漏算、错算或误解单位；而规则系统虽然可解释、稳定，但覆盖率有限，过度扩展又容易误伤。因此，本项目采取保守融合策略，只在规则或模板足够明确时覆盖模型输出，其余题目保留 few-shot CoT 的答案。

## 3. 方案一：Prompt/CoT 小模型直接推理

### 3.1 实现说明

Prompt 推理方案由 `qwen_prompt_infer.py` 实现。该脚本读取测试集 `test.json`，调用本地缓存的 `Qwen2.5-0.5B-Instruct` 模型完成生成，并将输出解析成提交 CSV。脚本支持三种主要推理策略：

```text
direct
zero_shot_cot
few_shot_cot
```

`direct` 策略要求模型直接输出答案，提示词尽量简短；`zero_shot_cot` 策略要求模型先进行简要推理，再输出“最终答案”；`few_shot_cot` 策略在 prompt 中加入若干训练集风格的示例，引导模型使用“分析过程 + 最终答案”的格式。推理后，脚本使用正则表达式和清洗逻辑从模型文本中抽取最终数字答案，避免解释文字进入提交文件。

实际运行中，few-shot CoT 明显优于 direct。其原因在于，小学应用题通常需要先理解题目关系，再列式计算。直接输出时，0.5B 模型容易只抓取题中某个数字或进行错误近似；加入少量示例后，模型更容易保持“读题、列式、计算、输出”的模式，格式稳定性也更好。

### 3.2 实验结果

Prompt 相关提交结果如下：

| 提交文件 | 方法 | 平台分数 |
| --- | --- | ---: |
| `submit_prompt_direct.csv` | 直接答案 prompt | 0.057500 |
| `submit_prompt_fewshot_cot.csv` | few-shot CoT prompt | 0.182125 |

可以看到，few-shot CoT 相比 direct prompt 有巨大提升。这说明对于小模型而言，提示词中的推理格式约束和少量示例非常关键。虽然 few-shot CoT 没有更新模型参数，但它已经成为所有后续混合方案的最佳基础答案来源。

### 3.3 分析

Prompt/CoT 路线的优点是实现成本低、复现简单，不需要训练；缺点是计算稳定性受模型能力限制明显。实际观察中，few-shot CoT 经常能识别题目大意，但在以下情形容易出错：

- 多步计算中只完成第一步，漏掉最后一步。
- 遇到“至少”“最多”时没有正确向上或向下取整。
- 单位换算不稳定，例如米、厘米、千克、吨、小时、分钟。
- 比例、分数、百分数题中选错基准量。
- 输出格式有时带解释，需要额外清洗。

因此，few-shot CoT 适合作为基础答案，但不适合作为最终唯一方案。后续规则求解器和模板校正层正是针对这些弱点设计的。

## 4. 方案二：数据构建与 LoRA SFT

### 4.1 数据增强实现

数据增强由 `augment_train.py` 实现。该脚本只使用训练集中的题目模式构造新样本，不引入外部题库或外部答案。增强样本主要覆盖以下题型：

- 简单四则应用题。
- 平均分、总量求部分、剩余量计算。
- “照这样计算”的单位率题。
- “最多/至少”的取整题。
- 简单倍数关系题。

脚本输出 `train_augmented.json`。当前生成的增强训练集约为 13999 条，其中包括原始训练集和约 2000 条模板合成样本。该方案的目的不是创造全新复杂知识，而是增加小模型对稳定算术模式的曝光次数，让模型在 SFT 后更倾向于输出规范数字答案。

### 4.2 CoT 数据构建实现

CoT 数据由 `build_cot_data.py` 实现。其思路是将训练题转换为带推理步骤的监督样本，统一格式为：

```text
解题步骤：...
最终答案：xxx
```

对于规则求解器可以解析的题目，脚本生成较具体的计算步骤；对于规则无法可靠解析但训练集中有答案的题目，脚本生成保守的通用步骤，强调“根据题意列式并核对答案”。最终输出 `train_cot.json`，包含约 5838 条 CoT 训练样本，其中规则型 CoT 约 1838 条，通用型 CoT 约 4000 条。

### 4.3 LoRA 训练与推理

LoRA 微调由 `qwen_lora_sft.py` 实现，推理由 `qwen_lora_infer.py` 实现。训练脚本支持设置训练文件、输出目录、答案字段、epoch、batch size、梯度累积和 bf16 等参数。推理脚本支持 direct 和 cot 两种模式，并在 CoT 模式下从“最终答案”字段抽取提交值。

典型训练命令如下：

```powershell
.\.venv\Scripts\python.exe qwen_lora_sft.py `
  --train train_augmented.json `
  --output-dir output/qwen_aug_sft `
  --epochs 3 `
  --batch-size 2 `
  --grad-accum 8 `
  --bf16
```

CoT 版本训练命令如下：

```powershell
.\.venv\Scripts\python.exe qwen_lora_sft.py `
  --train train_cot.json `
  --output-dir output/qwen_cot_sft `
  --answer-key cot_answer `
  --epochs 3 `
  --batch-size 2 `
  --grad-accum 8 `
  --bf16
```

本地运行环境为 RTX 4060 Laptop GPU，可以完成 0.5B 模型的 LoRA 训练和推理。训练结束后分别生成 `submit_aug_sft.csv` 和 `submit_cot_sft.csv`。

### 4.4 实验结果与分析

SFT 相关结果如下：

| 提交文件 | 方法 | 平台分数 |
| --- | --- | ---: |
| `submit_cot_sft.csv` | CoT 数据 LoRA SFT | 0.111875 |
| `submit_aug_sft.csv` | 增强数据 LoRA SFT | 0.133000 |

从结果看，SFT 方案没有超过 few-shot CoT。主要原因可能有三点。

第一，0.5B 模型容量有限，经过少量数据 SFT 后，模型可能更容易学习输出格式和浅层模板，而不是显著提升数学推理能力。第二，自动构造的 CoT 数据质量有限，部分通用步骤对模型学习帮助不大，甚至可能让模型学到“解释看似合理但计算不稳定”的模式。第三，训练集本身存在一定噪声，同题可能出现不同答案；小模型在 SFT 中会吸收这些噪声。

不过，SFT 探索仍然有价值。它验证了在本任务中，单纯增加训练并不一定优于设计良好的 prompt；同时也说明后续如果继续做，应重点提升 CoT 数据质量，而不是简单扩大模板样本数量。

## 5. 方案三：规则与模板求解器

### 5.1 基础规则求解器

规则求解器最初由 `math_solver.py` 实现，后续进一步扩展为 `improved_rule_hybrid.py`。规则系统的核心思想是将高频小学数学题型转为确定性计算。相比模型生成，规则的优点是可解释、稳定、不会出现格式漂移；缺点是覆盖范围有限，而且宽泛规则容易误伤。

规则求解器包含以下类型：

- 显式四则表达式，例如“两个数的和乘两个数的差”。
- 正方形、长方形、圆等基础几何公式。
- “至少需要几辆车/几条船”的向上取整。
- “最多可以买几个/做几件”的向下取整。
- 单位率和“照这样计算”题。
- 简单行程题、相向而行题。
- 分数、比例、百分率相关题。
- 训练/测试集中出现的高频同模板题。

为了避免低精度规则拉低分数，最终提交时只启用白名单中的高置信规则。每增加一个规则，都会先在训练集上统计覆盖数量和准确率，再检查它在测试集上实际覆盖的样例。对于存在明显歧义的规则，即使能覆盖更多题，也不进入最终提交。

### 5.2 改进规则混合

在 `improved_rule_hybrid.py` 中，项目逐步加入了更细的高置信题型，例如：

- `capacity_ceil`：车辆、船只、油桶等容量向上取整。
- `group_floor`：材料按每份用量最多可做多少份。
- `same_rate_distance`：同速度下距离与时间比例。
- `same_rate_food_yield`：原料和产物比例换算。
- `square_perimeter_direct`：正方形边长求周长。
- `square_side_from_perimeter`：正方形周长求边长。
- `same_rate_walk_time`：步行距离和时间的比例换算。
- `seeding_area`：播种宽度、速度和时间求面积。

规则版本从 v1 迭代到 v5。每一版都只增加少量人工检查过的高置信覆盖。v5 相比 few-shot CoT 改动 68 行，全部通过提交格式校验。虽然这个增量不可能带来大幅提升，但它为后续 v6 的最终融合提供了稳定基础。

### 5.3 训练集同题/同模板校正

在进一步分析中发现，测试集中存在一部分题目与训练集题干完全相同，或者数字不同但文字模板高度一致。由于只能使用训练集信息，这类样本可以作为高置信校正来源。项目新增 `build_final_hybrid.py`，在 v5 的基础上加入两类校正：

1. **训练集同题记忆**：如果测试题规范化后与训练题完全一致，且训练集中该题答案一致或存在多数答案，则使用训练答案覆盖模型输出。
2. **人工核验同模板校正**：对于训练集和测试集共享的模板，如果公式非常明确，则按测试题数字重新计算答案。

该层也进行了风险控制。训练集中存在少量噪声，例如同一道年龄题中训练答案可能没有考虑“明年”。因此实现中准备了 `memory` 和 `curated` 两个版本：`memory` 更接近平台标签，`curated` 排除人工发现的明显噪声。最终两个版本在平台上同分，说明该噪声样本没有影响公开计分结果。

### 5.4 最终融合流程

最终答案生成流程如下：

1. 以 `submit_prompt_fewshot_cot.csv` 作为基础答案。
2. 用 `improved_rule_hybrid.py` 中的高置信规则覆盖部分答案，生成 v5。
3. 用 `build_final_hybrid.py` 加入训练集同题记忆和同模板人工校正，生成 v6。
4. 对生成 CSV 进行严格校验：8000 行、无重复 id、答案非空、列数正确。
5. 提交 `v6_memory` 和 `v6_curated`，比较平台分数。

最终 v6 相比 few-shot CoT 共改动约 195 行。由于这些改动集中在高置信题上，平台分数从 `0.182125` 提升到 `0.194625`。

## 7. 实验分析

### 7.1 few-shot CoT 是最强基础方案

实验显示，在不训练或少训练的条件下，few-shot CoT 的效果优于 direct prompt 和两个 SFT 版本。其核心原因是小模型在示例约束下能更好地保持输出格式，也能更稳定地执行简单列式。direct prompt 虽然速度快，但经常只输出题目中的某个数字或做出浅层猜测。

这说明对于 0.5B 模型，prompt 设计本身就是重要工程。尤其在数学应用题任务中，示例不只是格式模板，也起到引导模型理解“题目 -> 算式 -> 答案”的作用。

### 7.2 SFT 没有带来预期提升

数据增强 SFT 和 CoT SFT 均低于 few-shot CoT。这个结果并不意味着 SFT 路线无效，而是说明本次自动构造数据的质量还不够高。增强数据虽然格式正确，但题型相对简单，可能导致模型更偏向模板记忆；CoT 数据虽然提供步骤，但一部分步骤较通用，对复杂题推理能力提升有限。

另外，小模型在 LoRA 训练后可能出现一定过拟合或格式迁移问题。例如模型更愿意输出训练集中常见的答案样式，但面对测试题中的新组合时仍然无法稳定计算。后续若继续优化，应考虑使用更高质量的 CoT 生成策略，或者构造“错误答案-正确答案”的偏好数据，让模型重点学习纠错。

### 7.3 规则系统适合做校正而不是全量求解

规则求解器在部分题型上非常有效，尤其是取整题、简单几何题、同速度比例题、训练集同模板题。但如果将规则写得过宽，会快速引入误伤。例如同样出现“比”“倍”“多”“少”的题目，可能是求某个对象数量，也可能是求总数或差值；同样出现“每”和“平均”的题目，也可能有多层单位换算。

因此最终方案采用“高置信白名单”策略，而不是全量规则覆盖。这个策略牺牲覆盖率换取准确率，更适合在提交机会有限的比赛场景中使用。

### 7.4 训练集同题和同模板利用是有效增量

v6 的主要提升来自训练集同题/同模板校正。该方法的工作量在于：

- 对训练集和测试集题干进行规范化，消除空格、全角符号等差异。
- 统计同题答案是否一致，排除明显冲突样本。
- 对同模板不同数字的题目进行人工检查，只保留公式明确的模板。
- 生成两个候选版本，对噪声题做风险隔离。

这一方法没有使用外部答案，而是充分利用训练集内部信息。它体现了比赛型任务中常见的数据分析思路：除了训练模型，还要理解数据分布、重复题、模板题和标注噪声。

### 7.5 为什么没有继续使用多数投票

本项目也比较过多个提交文件之间的答案差异。直觉上可以尝试 direct、zero-shot、SFT、few-shot 等多模型多数投票，但分析后没有采用。原因是多数投票在本任务中并不可靠。多个低分模型常常会在同一类浅层错误上达成一致，例如把“两边植树”算成单边数量，或者在分数题中选择错误基准。若直接使用多数投票，可能会覆盖掉 few-shot 或规则给出的正确答案。

因此，最终没有做全量 ensemble，而是只对可解释、可验证的规则题进行覆盖。这也是最终方案较保守但稳定的原因。

## 8. 创新性与工作量体现

本项目的创新性主要体现在以下几个方面。

第一，完成了多路线对比，而不是只提交单一 baseline。项目实现并实际运行了 prompt direct、few-shot CoT、数据增强 SFT、CoT SFT、规则求解和最终 hybrid，形成了完整实验闭环。

第二，提出了面向小模型弱点的分层校正框架。0.5B 模型负责提供大覆盖的基础答案，规则系统负责修正高置信算术题，训练集同题/同模板模块负责利用数据分布特征进行补强。三者职责清晰，避免了单一路线的短板。

第三，规则系统采用训练集准确率和人工样例检查相结合的方式逐步扩展。每个规则都经过覆盖统计和测试样例审查，不追求盲目扩大覆盖率。这种保守策略适合提交机会有限的场景。

第四，项目关注了真实数据中的噪声问题。训练集中存在同题不同答案和个别明显错误答案，最终实现中提供了 `memory` 与 `curated` 两个版本，用来比较“贴近训练标签”和“贴近数学题意”的差异。这体现了对数据质量和比赛评测机制的思考。

第五，完整实现了自动化生成与校验流程。所有提交 CSV 都经过行数、重复 id、空答案和格式校验，避免因格式问题损失分数。旧版本中曾发现部分输出存在格式异常，后续均增加了清洗和验证步骤。

## 9. 不足与后续改进方向

虽然最终方案相较 few-shot CoT 有稳定提升，但整体分数仍不高，说明系统仍有明显不足。

第一，复杂语义理解不足。很多题目需要理解隐含关系，例如年龄变化、工程合作、行程追及、比例尺、百分率基准等，仅靠当前规则和 0.5B 模型仍难稳定解决。

第二，CoT 数据质量有待提高。当前 CoT 数据部分由规则生成，部分为通用模板，缺少高质量、逐步可验证的推理链。后续可以使用更强模型在训练集上生成候选 CoT，再通过答案一致性和公式验证筛选，构造更可靠的 SFT 数据。

第三，规则覆盖仍然有限。最终为了避免误伤，只覆盖了少量高置信题。如果有更多时间，可以继续按题型簇分析训练/测试模板，扩展工程问题、百分数问题、比例尺问题、年龄问题和单位换算问题。

第四，缺少验证集驱动的系统调参。由于平台提交机会有限，项目主要依赖训练集统计和少量公开提交结果。后续可以从训练集中构造更合理的本地验证集，模拟 leaderboard，减少盲目提交。

第五，模型融合仍有空间。当前没有采用全量多数投票，因为低分模型错误相关性较高。后续可以训练一个轻量级答案选择器，输入题目、多个候选答案和规则特征，预测哪个候选更可信，而不是简单投票。

## 10. 结论

本项目围绕“小学数学应用题自动解题”完成了从小模型推理、数据构建、LoRA 微调、规则求解到最终混合提交的完整实验流程。实验表明，在 0.5B 模型限制下，few-shot CoT 是最强的基础方案，平台分数达到 `0.182125`；单纯的数据增强 SFT 和 CoT SFT 没有超过 few-shot，说明小模型数学推理能力和自动构造数据质量仍是瓶颈。

在此基础上，项目通过高置信规则求解和训练集同题/同模板校正进一步提升，最终提交 `submit_fewshot_improved_rule_hybrid_v6_memory.csv` 得分 `0.194625`。该结果较 few-shot CoT 净多答对约 100 题，验证了混合系统在小模型数学任务中的有效性。

从方法角度看，本项目最大的收获是：对于受限小模型，单纯“让模型学会全部数学推理”难度较高；更实际的路线是将模型的大覆盖语言理解能力、规则系统的精确计算能力、以及训练数据中的模板信息结合起来。最终方案虽然距离更高分仍有差距，但完成了多种方向的探索，具备可复现实现、明确实验结果和较完整的误差分析。
